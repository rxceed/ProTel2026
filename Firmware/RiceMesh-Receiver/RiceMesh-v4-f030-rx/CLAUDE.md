# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

Bare-metal STM32F030C8T6 (LQFP48) firmware that reads distance from an HC-SR04 ultrasonic sensor and transmits it over an nRF24L01+ radio. Generated by STM32CubeMX 6.16.1 (HAL driver framework, no RTOS), built with `arm-none-eabi-gcc` via Makefile.

## Build and flash

- `make` — produces `build/RiceMesh-v4-f030.{elf,hex,bin}`. Toolchain must be on `PATH` as `arm-none-eabi-*`, or pass `make GCC_PATH=/path/to/bin`.
- `make clean` — wipes `build/`.
- `./build_flash.sh` — builds, then flashes the ELF via OpenOCD + ST-Link using `bluepill.cfg`.
- Manual flash: `openocd -f bluepill.cfg -c "program build/RiceMesh-v4-f030.elf verify reset exit"`.
- `bluepill.cfg` uses `hla_swd` with `CPUTAPID 0x0bb11477` (STM32F0-series IDCODE) and `target/stm32f0x.cfg`.
- Debug build by default (`DEBUG = 1`, `-Og -g -gdwarf-2`). No release target, no tests, no linter.

## Architecture

### Code generation boundary (important)

`RiceMesh-v4-f030.ioc` is the CubeMX project file. Regenerating from it **overwrites** `Core/`, `Drivers/STM32F0xx_HAL_Driver/`, `Drivers/CMSIS/`, `Makefile`, `STM32F030XX_FLASH.ld`, and `startup_stm32f030x8.s`. Hand-written application code must live **between `/* USER CODE BEGIN x */` and `/* USER CODE END x */` markers** in the generated files (mainly `Core/Src/main.c`, `Core/Inc/main.h`, `Core/Src/stm32f0xx_it.c`) — everything else is regenerated and lost.

Custom drivers (`Drivers/nRF24/`) sit outside the CubeMX-managed tree, are safe to edit freely, and **must be added to `C_SOURCES` in the Makefile** since CubeMX doesn't know about them.

### .ioc / source drift (active mismatch)

The `.ioc` and the generated sources are out of sync — the `.ioc` was updated (PA8 reassigned to `TIM1_CH1` input-capture for the echo) but CubeMX was never re-run. Symptoms in the tree:

- `MX_GPIO_Init` references `ECHO_Pin` / `ECHO_GPIO_Port` macros that are **not defined anywhere** — the build fails until these are added (or the stale block is removed).
- `MX_TIM1_Init` only initializes the time base; there is no `HAL_TIM_IC_ConfigChannel` call, yet `main.c` calls `HAL_TIM_IC_Start_IT(&htim1, TIM_CHANNEL_1)` and relies on `HAL_TIM_IC_CaptureCallback`.
- PA8 (the new echo pin per `.ioc`) is never configured as GPIO/AF.

Before doing significant work, either regenerate from CubeMX or manually reconcile the two. Don't assume the C source reflects current intent.

### Application

Single-threaded super-loop in `Core/Src/main.c`:

1. `HCSR04_Read()` (USER CODE 0): pulses `TRIG_Pin` (PB1) high for ~10 TIM1 ticks via `delay()` (busy-wait on the counter), then enables `TIM_IT_CC1`. The IC callback captures rising then falling edge, computes `Difference` (timer ticks), stores `Distance = Difference * 0.034 / 2` (cm), and disables the interrupt. **No timeout** if the echo never arrives.
2. Main `while(1)`: `HCSR04_Read()` → `HAL_Delay(60)` to let the capture finish → format `"N2 D:%u cm"` into `data_T[32]` → `nrf24_transmit()`; on success (return 0) pulse PB5 LED on for 50 ms then off → `HAL_Delay(500)`.

`Distance` is `uint8_t`, so readings above 255 cm wrap silently even though HC-SR04 reaches ~400 cm.

### TIM1 sharing and the 1 µs assumption

TIM1 has three roles, all depending on the counter ticking at **1 MHz** (1 µs/count):
- HC-SR04 distance formula (`Difference * 0.034 / 2` is only correct if `Difference` is in µs).
- `delay()` in `main.c`.
- `delay_us()` inside the nRF24 driver, used for CE-pulse timing.

`MX_TIM1_Init` currently sets `Prescaler = 47`. For a 1 MHz tick that requires a 48 MHz timer clock, but `SystemClock_Config()` actually delivers `SYSCLK = 16 MHz` (HSI 8 MHz → /2 → ×PLLMUL4), so the timer ticks at 16/(47+1) ≈ 333 kHz (3 µs/count). Net effect: reported `Distance` and every `delay_us()` are roughly 3× off. Fix is either bumping PLL to 48 MHz (PLLMUL12, `FLASH_LATENCY_1`) or dropping the prescaler to 15.

The IC measurement and the nRF24 driver share TIM1's counter, so they cannot run concurrently — the loop sequences them. Anything new that calls `delay_us()` while a capture is pending will corrupt one or the other.

### nRF24 driver wiring

Configured entirely through `Drivers/nRF24/NRF24_conf.h`, which references HAL handles by name:

- `hspiX` → `hspi1`
- `htimX` → `htim1` (used for `delay_us`)
- `csn_gpio_*`, `ce_gpio_*` → PA3 / PA4

If SPI, timer, or CSN/CE assignments change in CubeMX, **`NRF24_conf.h` must be updated to match** — there is no compile-time check.

Vendor-driver quirk already patched: `nrf24_transmit_no_ack` had a header-vs-implementation return-type mismatch (header `uint8_t`, body `void` with bare `return 1;`) that broke compilation. The definition was fixed locally; re-apply if the driver is ever re-vendored.

### Pin map (per `.ioc`; see drift note above)

| Pin   | Function                                    |
|-------|---------------------------------------------|
| PA3   | nRF24 CSN (output)                          |
| PA4   | nRF24 CE  (output)                          |
| PA5-7 | SPI1 SCK/MISO/MOSI                          |
| PA8   | TIM1_CH1 input capture — HC-SR04 ECHO       |
| PA9/10| USART1 TX/RX (38400 8N1, unused in app)     |
| PA0   | EXTI0 input, falling edge, no handler logic |
| PB1   | HC-SR04 TRIG (output, macro `TRIG_Pin`)     |
| PB5   | TX-success LED — no symbolic name, written as raw `GPIO_PIN_5` |

On Cortex-M0, EXTI lines 0 and 1 share a single `EXTI0_1_IRQn` vector (unlike F103's per-line vectors).

### Clock tree

`SystemClock_Config()` runs from **HSI** (8 MHz internal RC). On F0 with `PLLSource = HSI`, PLL input is HSI/2 = 4 MHz, ×PLLMUL4 → SYSCLK = 16 MHz, AHB/APB ÷1, `FLASH_LATENCY_0`. No external crystal. See the TIM1 section for the timing consequence.

### Error handling

`Error_Handler()` disables interrupts and spins forever — no recovery path, no diagnostic output. USART1 is initialized but `printf` is not wired to it.

# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

Bare-metal STM32F103C8T6 (Blue Pill) firmware that reads distance from an HC-SR04 ultrasonic sensor and transmits it over an nRF24L01+ radio. Project was generated by STM32CubeMX 6.16.1 (HAL driver framework, no RTOS) and built with `arm-none-eabi-gcc` via Makefile.

## Build and flash

- `make` — builds `build/RiceMesh-v2.{elf,hex,bin}`. Toolchain must be on `PATH` as `arm-none-eabi-*`, or pass `make GCC_PATH=/path/to/bin`.
- `make clean` — wipes `build/`.
- `./build_flash.sh` — builds, then flashes the ELF via OpenOCD + ST-Link using `bluepill.cfg` (`hla_swd`, CPUTAPID `0x2ba01477` — the clone-chip ID; do not change unless using a genuine STM32). Aborts flashing if build fails.
- Manual flash (without the script): `openocd -f bluepill.cfg -c "program build/RiceMesh-v2.elf verify reset exit"`.
- Debug build is on by default (`DEBUG = 1`, `-Og -g -gdwarf-2` in the Makefile). There is no separate release target.
- No tests, no linter — this is a small embedded project.

## Architecture

### Code generation boundary (important)

`RiceMesh-v2.ioc` is the CubeMX project file. Regenerating from it **overwrites** `Core/`, `Drivers/STM32F1xx_HAL_Driver/`, `Drivers/CMSIS/`, `Makefile`, `STM32F103XX_FLASH.ld`, and `startup_stm32f103xb.s`. Hand-written application code must live **between `/* USER CODE BEGIN x */` and `/* USER CODE END x */` markers** in the generated files (mainly `Core/Src/main.c`, `Core/Inc/main.h`, `Core/Src/stm32f1xx_it.c`) — everything else will be regenerated and lost.

Custom drivers (`Drivers/nRF24/`) sit outside the CubeMX-managed tree and are safe to edit freely, but their C sources must be added to `C_SOURCES` in the Makefile. CubeMX **does not** know about them.

### Application

Single-threaded super-loop in `Core/Src/main.c`:

1. `HCSR04_Read()` (defined in `main.c` USER CODE 0) — drives `TRIG_Pin` (PB1) for 10 µs, busy-waits for `ECHO_Pin` (PB0) rising edge, then uses **TIM1** as a free-running 1 MHz microsecond counter (prescaler 15 against the 16 MHz APB2 timer clock) to measure echo pulse width. Returns distance in cm with a ~6.5 m timeout. **TIM1 is shared with the nRF24 driver's `delay_us()`** — they cannot run concurrently.
2. Main loop reads distance, formats it as ASCII into a 32-byte payload, and pushes it through `nrf24_transmit()`. LED on PB5 lights during TX; loop period is `HAL_Delay(500)`.

### nRF24 driver wiring

The radio driver (`Drivers/nRF24/NRF24.c` / `.h`) is configured purely through `Drivers/nRF24/NRF24_conf.h` — it references the HAL handles by name:

- `hspiX` → `hspi1` (CubeMX-generated SPI handle)
- `htimX` → `htim1` (used for `delay_us()`)
- `csn_gpio_*`, `ce_gpio_*` → PA3 / PA4

If you change the SPI peripheral, timer, or CSN/CE pin assignment in CubeMX, **`NRF24_conf.h` must be updated to match** — there is no compile-time check that the names line up with what CubeMX actually generated. The driver assumes the timer is configured for 1 MHz tick (1 µs per count).

### Pin map (see `Core/Inc/main.h` and `RiceMesh-v2.ioc`)

| Pin   | Function                                  |
|-------|-------------------------------------------|
| PA3   | nRF24 CSN (output)                        |
| PA4   | nRF24 CE  (output)                        |
| PA5-7 | SPI1 SCK/MISO/MOSI                        |
| PA9/10| USART1 TX/RX (115200 8N1, currently unused in app logic) |
| PA0   | EXTI0 input (falling edge IRQ enabled, no handler logic yet) |
| PB0   | HC-SR04 ECHO (input)                      |
| PB1   | HC-SR04 TRIG (output)                     |
| PB5   | TX-activity LED                           |

### Clock tree

`SystemClock_Config()` runs from **HSI** (8 MHz internal RC), PLL ×4 against HSI/2 → SYSCLK = 16 MHz, all bus prescalers ÷1. No external crystal is configured. `FLASH_LATENCY_0` is fine at this frequency. Anything timing-sensitive (especially `delay_us()` and the HC-SR04 echo measurement) assumes 16 MHz SYSCLK and TIM1 prescaler 15 → 1 MHz.

### Error handling

`Error_Handler()` in `main.c` disables interrupts and spins forever — there is no recovery path or diagnostic output. If you need debug telemetry, USART1 is initialized but unused; `printf` is not wired to it.
